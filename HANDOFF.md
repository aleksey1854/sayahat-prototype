## 2. УСЛОВИЯ СДЕЛКИ И ГРАНИЦЫ ЭТАПОВ (важно — деньги)
═══════════════════════════════════════════════════════════════════════

Виктор своими словами разбил на 3 этапа (скрин переписки был):
- **Этап 1** — сайт-мультикаталог. ~300к ₸, ~3 недели. Наполняет контентом сам
  исполнитель (Виктор подтвердил письменно). «На этом этапе тестируем — если есть
  эффект, идём дальше».
- **Этап 2** — витрины для участников (личные кабинеты арендаторов + чат-бот).
- **Этап 3** — онлайн-продажи + доставка.

КРИТИЧНО ПРО ГРАНИЦУ 1↔2: был момент, когда Виктор написал «мультикаталог = то же,
что мультилендинг», и исполнитель в моменте согласился «тогда добавляем личные
кабинеты сейчас». Это ядро Этапа 2 (за отдельные ~300к). Позже по переписке стало
видно, что Виктор сам кладёт «витрины для участников» во 2-й этап. Позиция, которую
держим: **кабинеты/админка построены ДОСРОЧНО как задел Этапа 2 и как инструмент
наполнения для Этапа 1; при сдаче показываем их Виктору как живое демо Этапа 2**,
а не «бесплатно отдали в Этап 1». Это защищает ценность Этапа 2 (там остаётся
чат-бот, модерация, привлечение). Если Виктор снова смешивает этапы — фиксировать
состав и сроки письменно, вежливо и по его же логике.

«Эффект» для Виктора = звонки и клики покупателей. Поэтому в проекте есть цели
аналитики (call/whatsapp/kaspi) — через месяц после запуска это аргумент продать
Этапы 2–3 цифрами.

## 3. СТЕК И АРХИТЕКТУРА
═══════════════════════════════════════════════════════════════════════

Next.js 14.2.35 (App Router) + TypeScript + Tailwind (только базовые утилиты — без
кастомного конфига) + Prisma 5.22. React 18. Шрифты Onest+Manrope самохостятся
(@fontsource, НЕ Google CDN). Иконки — инлайн-SVG.

БАЗА: **PostgreSQL** (Neon через Vercel). Env: DATABASE_URL (pooled, с `-pooler`) +
DATABASE_URL_UNPOOLED (direct, для миграций). Локальная разработка идёт против
ОБЛАЧНОЙ базы — те же строки в локальном .env. (Раньше был SQLite — мигрировали.)

ФОТО: **Vercel Blob** в проде / диск (папка uploads/) локально. Переключается по
наличию BLOB_READ_WRITE_TOKEN. Файлы: {id}.webp (до 1600px) + {id}-sm.webp (640px,
для карточек, через srcset). Поворот по EXIF, метаданные вычищаются, лимит 8МБ,
HEIC ловится и отклоняется с инструкцией.

ХОСТИНГ: **Vercel** (задеплоено, работает). Деплой из GitHub, автосборка по push.
Build = `prisma generate && next build`. Домен пока `*.vercel.app`.

Схема БД (prisma/schema.prisma): Category(slug, nameRu/Kz, order) · Shop(slug,
nameRu/Kz, descRu, categoryId, cover, phone, whatsapp, instagram, kaspiUrl, row,
pavilion, landmark, hours, rating, layout=JSON-строка{tagline,about,promo,gallery},
metaTitle/Desc, status=draft|published) · Product(nameRu/Kz, descRu, price, oldPrice,
unit, image, order) · NewsPost · Account(login unique, passwordHash bcrypt,
role=tenant|admin, shopId unique).

Структура src — см. дерево в конце файла и README.md.

## 4. ЧТО УЖЕ СДЕЛАНО (Этап 1 + задел Этапа 2)
═══════════════════════════════════════════════════════════════════════

КАТАЛОГ (главная, src/app/page.tsx + CatalogSection): поиск (в шапке), фильтр по
категориям, сетка магазинов, реальные счётчики, новости (если есть), карта рядов.

СТРАНИЦА МАГАЗИНА (shop/[slug]): герой, «о магазине», товары с ценами и скидками,
акция, галерея, прайс-лист, контакты (звонок/WhatsApp/Instagram/Kaspi), «как найти»
с подсветкой ряда на схеме, липкая панель звонка на мобиле, черновик/публикация.

ПОИСК (src/lib/search.ts) — доведён до предела, 49 тестов (`npm run test:search`):
полный русский стеммер Snowball (падежи/числа/прилагательные/причастия) + казахские
окончания + уменьшительные/беглые гласные + синонимы RU↔KZ (synonyms.ts) +
направленные гиперонимы + опечатки (в т.ч. с падежом и в недописанном слове) +
раскладка + транслит + слитные написания + фразовый бонус. Слабые совпадения
(опечатки/склейки) показываются ТОЛЬКО когда точных нет (иначе «курага» тянула
«галантерею»). Под полем — выпадающие подсказки (превью, категория·ряд, товар с
ценой, «Показать все», клавиатура ↑↓/Enter/Esc). Аналитика запросов: события
search / search_empty.

SEO (код-потолок): мета + OG (og.png 1200×630) + микроразметка (ShoppingCenter,
Store с priceRange и sameAs, BreadcrumbList, ItemList товаров с ценами KZT) +
каноникалы + sitemap.xml (из БД) + robots.txt (закрыты /admin /cabinet /login) +
авто-описания магазинов из товаров + verification-коды через env. Чек-лист проверки
и словарик терминов — в README.

ФОТО (src/lib/img.ts + imageMeta.ts + PhotoInput.tsx) — 21 тест (`npm run test:images`):
два размера + srcset, живой предпросмотр «как ляжет на карточку», определение
соотношения (1:1/4:3/9:16…), предупреждения под место (обложка/товар: вертикальное/
панорама/мелкое/сторис), HEIC-инструкция, отчёт о сжатии («сжато до N КБ»).

ДВА ЯЗЫКА RU/KZ (src/lib/i18n.ts, cookie, один адрес): каталог, лендинги, карта,
футер, 404. NB: казахские тексты писал AI без носителя — перед сдачей дать вычитать.

КАБИНЕТЫ И АДМИНКА (задел Этапа 2, но РАБОТАЕТ и служит наполнению Этапа 1):
- /login — вход (rate-limit 10/15мин, timing-safe bcrypt, редирект по роли).
- /cabinet — арендатор правит свою страницу: названия, описание, «о магазине»,
  контакты, место, часы, обложку, товары (CRUD, порядок ↑↓, фото). Нормализация
  телефонов (8→+7), WhatsApp (только цифры), защита kaspiUrl от javascript: (XSS).
- /admin — создать магазин (черновиком, slug транслитом вкл. каз. буквы), выдать/
  сменить логин арендатору, публикация, удаление (чистит фото+доступ), категории
  (add/rename/порядок/delete-if-empty), новости, «Редактировать» = impersonate.
- Демо-доступы из сида: admin/admin2026, dastarkhan/demo2026. Новый админ:
  `npm run admin:create -- логин пароль`.

ТЕХ-КАЧЕСТВО: next обновлён до 14.2.35 (закрыта уязвимость), SubmitButton (состояние
«Сохраняю…» + защита от двойного клика), фокус-рамки, alt-тексты, брендовая
страница ошибки, lazy-загрузка фото.

## 7. ГРАБЛИ И ПРАВИЛА РАБОТЫ (реально возникали)
═══════════════════════════════════════════════════════════════════════

ОБНОВЛЕНИЕ ПРОЕКТА ИЗ АРХИВА (частая боль): распаковывать НЕ заменой всей папки.
Копировать только src/, prisma/, scripts/, package.json, *.md (и public/ при смене
картинок). НЕ трогать node_modules/, .env, папку uploads/ и облачную базу.
- Если менялся package.json — ОБЯЗАТЕЛЬНО `npm install` перед `npm run dev`
  (иначе «'next' is not recognized» или «Module not found @fontsource/...»).
- Если менялась схема — `npx prisma db push` на облачную базу.
Эти две ошибки уже ловили вживую — почти всегда причина в пропущенном npm install
или db push.

ДЕПЛОЙ VERCEL (выполнен, детали в VERCEL.md): сначала создать Neon Postgres + Blob,
env-переменные (DATABASE_URL, DATABASE_URL_UNPOOLED, BLOB_READ_WRITE_TOKEN, SITE_URL,
SESSION_SECRET), наполнить базу локально (пишет в облако), затем Redeploy. `db:seed`
стирает всё — НЕ запускать после начала реального наполнения.

БЕЗОПАСНОСТЬ: исполнитель однажды выложил строки подключения Neon и Blob-токен
ОТКРЫТЫМ ТЕКСТОМ в чат. НАДО НАПОМНИТЬ их ротировать (Neon: reset password;
Vercel Blob: перевыпустить токен; обновить env на Vercel и локально). НЕ СДЕЛАНО —
напомнить при удобном моменте.

СТИЛЬ РАБОТЫ, который зашёл исполнителю: короткие ответы по делу, без воды и лести;
готовые файлы целиком, не сниппеты; честно проговаривать границы («это код-потолок,
дальше только данные/дизайн»); не бросаться на крупную переделку, если она пересекается
с грядущим решением Виктора; каждую сборку проверять (tsc --noEmit + next build +
тесты) перед отдачей; отдавать перепакованный ZIP через outputs.

ПРОВЕРКА СБОРКИ В ПЕСОЧНИЦЕ: Prisma engine в песочнице не работает — ставятся заглушки
(PRISMA_ENGINES_CHECKSUM_IGNORE_MISSING=1 + пустышки libquery_engine и schema-engine
в node_modules/@prisma/engines), проверяется только tsc + next build (страницы
динамические из-за cookies, к БД при сборке не ходят). Тесты поиска/фото гоняются tsx.
