# Обновление базы на проде (Neon + Vercel)

Конфигурация проекта: PostgreSQL на Neon, подключение через Vercel.
В `prisma/schema.prisma` два адреса: `DATABASE_URL` (pooled, для сайта) и
`DATABASE_URL_UNPOOLED` (direct, для операций со схемой и сида).

---

## Главное, что нужно понимать

`npm run db:seed` — **не «дозалить новое», а «стереть и залить заново»**.
Первым делом он выполняет:

```
account.deleteMany()   ← аккаунты: админ и арендаторы
product.deleteMany()   ← товары и цены
shop.deleteMany()      ← магазины
category.deleteMany()  ← категории
newsPost.deleteMany()  ← новости
```

То есть сотрёт и всё, что Виктор или арендаторы завели через `/admin` и `/cabinet`,
включая пароль админа (после сида он вернётся к демо-значению `admin / admin2026`).

Сейчас на проде лежит старое демо (магазины «Дастархан», «Специи Востока»,
`pavilion` = «14», `row` = «Б»), поэтому терять там нечего — но проверь это
перед запуском, а не после.

---

## Перед началом: есть ли что терять

Открой на проде:

- `/admin` → список магазинов. Если там всё ещё старое демо (9 магазинов
  с адресами вида «14 · бутик №Б») — реального контента нет, сид безопасен.
- Если появились реальные арендаторы, их товары или загруженные фото —
  **не запускай сид**, переходи в раздел «Точечное обновление» ниже.

---

## Бэкап (1 минута, делай всегда)

Neon умеет откатываться сам, но проще всего сделать ветку:

1. Neon → проект → **Branches** → **New branch**.
2. Назови, например, `before-seed-2026-07`.
3. Создать.

Это моментальный снимок. Если что-то пойдёт не так, данные можно достать
из ветки. Ветку потом удалишь.

---

## Обновление данных

### 1. Возьми прод-строки подключения

Vercel → проект → **Settings** → **Environment Variables**.
Нужны две:

- `DATABASE_URL` — та, где в адресе есть `-pooler`
- `DATABASE_URL_UNPOOLED` — без `-pooler`

### 2. Пропиши их локально

В `.env` в корне проекта:

```
DATABASE_URL="postgresql://…-pooler…?sslmode=require"
DATABASE_URL_UNPOOLED="postgresql://…?sslmode=require"
```

Убедись, что это именно **прод**-база, а не тестовая: сид сотрёт то, куда
подключишься.

### 3. Если менялась схема

Сейчас — **не менялась**, этот шаг пропускаешь. Он нужен, только если в
`prisma/schema.prisma` появились новые поля:

```bash
npm run db:push
```

### 4. Залей данные

```bash
npm install       # только если ставишь проект заново
npm run db:seed
```

В базе окажется: 15 категорий, 50 реальных арендаторов, 3 новости,
демо-доступы `admin / admin2026` и `ayanet / demo2026`.

### 5. Смени пароль админа

Сид создаёт демо-админа. Сразу после заливки:

```bash
npm run admin:create -- своелогин свойпароль
```

Логин и пароль передаются аргументами (двойное тире обязательно — иначе npm
съест их сам). Демо-аккаунт `admin / admin2026` после этого удали через `/admin`.

### 6. Проверь сайт

Открой прод и убедись:

- на карточках адрес вида «Продуктовый · бутик №14», а не «14 · бутик №Б»;
- под категориями появился ряд «Все павильоны · Продуктовый · Вещевой №1 · Вещевой №2»
  (до заливки он скрыт, потому что павильоны не заполнены);
- на странице магазина его павильон подсвечен на схеме зелёным;
- новости — «Рынок «Саяхат» теперь онлайн», а не «Базар Саяхат».

Редеплой не нужен: страницы с базой динамические, данные подтягиваются на
каждый запрос.

### 7. Убери прод-строки из локального `.env`

Чтобы случайно не запустить сид ещё раз по рабочей базе. Верни туда адрес
тестовой базы или закомментируй строки.

---

## Если контент уже есть — сид запускать нельзя

И тут есть неприятный нюанс, о котором лучше знать заранее.

Поля магазина (павильон, бутик, телефон, часы) редактируются **только в
`/cabinet`, под аккаунтом самого магазина**. Админ туда зайти не может:
кабинет жёстко привязан к аккаунту, и админа он отправляет обратно в `/admin`.
Админка правит только slug, порядок и аккаунты — самих полей магазина там нет.

То есть поправить павильоны у 50 магазинов руками = создать каждому аккаунт
через `/admin` и зайти под каждым. Это нереально.

Практический вывод:

- **Пока на проде демо — заливай сидом.** Это единственный дешёвый момент.
- **Когда появится реальный контент** — нужен отдельный скрипт, который
  обновит только `pavilion` и `row` у существующих магазинов по slug и ничего
  не удалит. Скажи — напишу, это недолго.

Товары и цены в любом случае заводят сами арендаторы через `/cabinet`.

---

## Чего не делать

- **`npm run db:reset` на проде.** Это `prisma db push --force-reset` —
  дропает и пересоздаёт всю схему, а не только чистит таблицы. На рабочей
  базе это лишнее.
- **Запускать сид дважды подряд «на всякий случай».** Каждый запуск —
  новое стирание.
- **Держать прод-строки в `.env` постоянно.** Одна невнимательная команда —
  и база чистая.

---

## Мелочь по проекту

`prisma/dev.db` — файл от старой SQLite-версии. Схема давно на PostgreSQL
(`provider = "postgresql"`), Prisma к этому файлу не обращается. Его можно
удалить, на работу это не влияет. В архив проекта он и так не попадает.

Ещё: если Виктор загружал фото через админку, файлы лежат в Vercel Blob.
Сид стирает записи магазинов, но сами файлы в Blob остаются — просто
превращаются в неиспользуемый мусор. На работу сайта не влияет, но при
большом объёме их стоит подчистить в панели Blob.
