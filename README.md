# Саяхат — сайт-мультикаталог (Этап 1)

Онлайн-каталог базара Саяхат (Костанай): каталог магазинов с поиском и фильтром,
лендинги магазинов, базовое SEO. Next.js 14 (App Router) + TypeScript + Tailwind + Prisma.
Дизайн перенесён из прототипа один-в-один.

## Требования

Node.js 18.18+ (лучше 20+).

## Запуск

```bash
npm install          # ставит зависимости и генерирует Prisma Client (postinstall)
npx prisma db push   # создаёт dev.db по схеме
npm run db:seed      # наполняет БД данными из прототипа: 6 категорий, 9 магазинов, товары, новости
npm run dev          # http://localhost:3000
```

Открой http://localhost:3000 — увидишь каталог. Магазины по ссылкам, например /shop/dastarkhan.

## Команды

- `npm run dev` — дев-сервер
- `npm run build` / `npm start` — прод-сборка и запуск
- `npm run db:seed` — заполнить БД
- `npm run db:reset` — пересоздать БД с нуля и заполнить заново

## Структура

```
prisma/
  schema.prisma      модель данных (Category, Shop, Product, NewsPost)
  seed.ts            данные из прототипа
src/
  app/
    layout.tsx       корневой layout, шрифты, метаданные
    page.tsx         главная — каталог
    shop/[slug]/     лендинг магазина (SSR + SEO + JSON-LD)
    not-found.tsx    404
    globals.css      дизайн-система из прототипа + Tailwind
  components/        Header, Footer, CatalogSection, BazaarMap, CallBar, LangToggle
  lib/               db (Prisma), i18n, site (контакты базара), format
```

## База данных

База — PostgreSQL (Vercel Postgres / Neon). Строки подключения — в `.env`:
`DATABASE_URL` (пул, для приложения) и `DATABASE_URL_UNPOOLED` (прямое, для миграций).
Локальная разработка ведётся против облачной базы — те же строки в локальном `.env`.

Данные лендинга сверх товаров (обложка «о магазине», акция, галерея, преимущества)
хранятся в поле `Shop.layout` (JSON) — это ядро будущего конструктора на Этапе 2.

Создать/обновить таблицы: `npx prisma db push`. Демо-данные: `npm run db:seed`
(стирает всё и заливает демо — не запускать после начала реального наполнения).

## Наполнение реальными данными

Встроенная админка БД — Prisma Studio:

```bash
npx prisma studio
```

Откроется http://localhost:5555 — таблицы Shop / Product / Category / NewsPost, можно
добавлять и править строки как в таблице, без кода. Порядок:

1. Фото магазина положить файлом в `public/photos/` (латиницей, например `magazin-akmaral.jpg`).
2. В Studio создать Shop: `slug` (латиницей, попадёт в адрес `/shop/slug`), названия RU/KZ,
   категория, `cover` — имя файла фото, телефон/WhatsApp (в формате `77051234567`), ряд/павильон,
   `status` = `published`.
3. Добавить Products магазина (связь по shopId, там же в Studio).
4. Поле `layout` можно не трогать — страница соберётся из товаров и контактов;
   «о магазине»/акцию/галерею заполним по образцу «Дастархана» при необходимости.

`npm run db:seed` НЕ запускать после начала реального наполнения — он стирает всё и заливает демо.

## Кабинет арендатора

- `/login` — вход (логин/пароль). `/cabinet` — редактирование своей страницы:
  название RU/KZ, описание, подзаголовок, «О магазине», контакты, место, часы.
- Доступ выдаёт админ со страницы `/admin` (см. ниже).
- Демо-доступ из сида: логин `dastarkhan`, пароль `demo2026`.
- Товары в кабинете: добавление, правка (цена, старая цена для скидки, единица, описание),
  фото, порядок кнопками ↑↓, удаление.
- Фото: при выборе файла — живой предпросмотр «как ляжет на карточку», размеры,
  соотношение (1:1, 4:3, 9:16…) и предупреждения под конкретное место: вертикальное
  для обложки, панорама, слишком мелкое, формат «сторис»; HEIC с iPhone ловится
  с инструкцией, что делать. При загрузке делаются два размера (1600px и лёгкий
  640px для карточек — браузер берёт нужный через srcset), поворот по EXIF
  чинится, метаданные вычищаются, итоговый вес виден в уведомлении («сжато
  до N КБ»). Хранилище переключаемое: с `BLOB_READ_WRITE_TOKEN` фото уходят
  в Vercel Blob (прод), без него — на диск в `uploads/` (локально, в git не
  попадает). Демо-фото — именем из `public/photos`. Тесты конвейера:
  `npm run test:images` (21 проверка: размеры, версии, коды ошибок, соотношения).

## Админка

- `/admin` (роль admin): создать магазин (появляется черновиком), выдать или сменить
  логин/пароль арендатора, «Редактировать» (открывает кабинет этого магазина),
  опубликовать/снять с публикации, удалить (с галочкой-подтверждением), категории
  (добавить, переименовать RU/KZ, порядок ↑↓, удалить пустую), новости главной.
- Черновики видны только админу и владельцу магазина (с плашкой «Черновик»),
  в каталог и sitemap не попадают.
- Рабочий цикл: создать магазин → создать доступ → «Редактировать» и наполнить → «Опубликовать».
- Демо-админ из сида: логин `admin`, пароль `admin2026`.
- Добавить админа в живую базу без пересоздания: `npm run admin:create -- логин пароль`
  (тем же скриптом меняется пароль существующего).

## Поиск

Свой движок (`src/lib/search.ts`), работает мгновенно на клиенте.

- Полная русская морфология — стеммер Snowball (тот же алгоритм, что внутри
  Elasticsearch и PostgreSQL для русского): падежи и числа существительных
  («орехами», «курагу»), прилагательные («узбекской» находит «узбекская»),
  причастия. Плюс слой для уменьшительных и беглых гласных:
  платок/платки/платочки, огурец/огурцы/огурчики, помидоры/помидорчики.
- Синонимы двух видов: переводы русский↔казахский («ет» находит «мясо»,
  «орамал» — «платок» — даже если магазин заполнен только по-русски) и
  направленные гиперонимы («изюм» находит лавку сухофруктов, «овощи» — все
  овощи, но «помидоры» не находят «огурцы»). Словарь: `src/lib/synonyms.ts`,
  правится руками под ассортимент.
- Казахские окончания: «жаңғақты» находит «жаңғақтар» (мн. число, падежи).
- Опечатки («куррага»), в том числе вместе с падежом («куррагой») и в
  недописанном слове («курог»); слитные написания («рахатлукум», «аяккиим»);
  не та раскладка («rehfuf» = «курага»), латиница-транслит («kuraga»),
  казахские буквы без спецсимволов («таттилер» = «тәттілер»),
  многословные запросы, поиск по товарам.
- Ранжирование: имя > категория > товар > описание; точное совпадение выше
  формы слова, форма — выше опечатки. На карточке показывается найденный товар
  с ценой; если совпали не все слова — честная пометка «показываем похожие».
  Слабые совпадения (опечатки, склейки слов) показываются ТОЛЬКО когда точных
  нет вообще — точная «курага» не тащит за собой галантерею. Под обоими полями
  поиска — выпадающие подсказки: превью, категория·ряд, найденный товар с ценой,
  «Показать все», стрелки/Enter/Esc.
  Фраза, встретившаяся подряд («грецкий орех» в названии товара), ранжируется
  выше тех же слов, разбросанных по разным полям.

Тесты: `npm run test:search` (49 проверок: морфология, синонимы и кросс-язык,
опечатки, раскладка, казахский, ранжирование, защита от ложных срабатываний).

## SEO: что сделано и как проверить

Сделано: meta + Open Graph на каждой странице, каноникалы, sitemap.xml
(обновляется сам), robots.txt (служебные страницы закрыты), микроразметка
Schema.org — ShoppingCenter (базар), Store (магазин, со ссылкой на Instagram),
BreadcrumbList (хлебные крошки), ItemList из Product с ценами в тенге
(шанс на расширенные сниппеты: цена и наличие прямо в выдаче Google),
lazy-загрузка фото, alt-тексты. Шрифты самохостятся (ноль запросов к Google —
быстрее первая отрисовка), OG-картинка 1200×630 (`public/og.png`) для превью
ссылок в WhatsApp/Telegram, описания магазинов собираются из товаров
автоматически (если поле metaDesc не заполнено), priceRange магазина
в разметке, коды подтверждения Search Console и Вебмастера — через `.env`
(`GOOGLE_SITE_VERIFICATION` / `YANDEX_VERIFICATION`). Отдельных /kz-адресов
и hreflang нет осознанно: язык переключается на одном адресе через cookie.

Словарик — что это всё такое, простыми словами:

- Мета-теги (title и description) — заголовок и серый текст сниппета в выдаче
  Google/Яндекса. Title влияет на позицию по словам, description — на то,
  кликнут ли. У каждого магазина свои; description собирается из его товаров.
- Open Graph — карточка, которая разворачивается при отправке ссылки
  в WhatsApp/Telegram: картинка + заголовок + описание.
- Микроразметка (JSON-LD) — те же данные на языке роботов: невидимый блок
  в коде, откуда поисковик достаёт цены, крошки, телефон. За это дают
  расширенные сниппеты.
- Каноникал — тег «главный адрес страницы»: защищает от дублей (utm-метки,
  RU/KZ на одном адресе) — весь ссылочный вес склеивается в один URL.
- Sitemap — список страниц для робота; robots.txt — куда роботу можно/нельзя.

Чек-лист проверки качества:

До деплоя (локально):
1. Открой страницу магазина → «Просмотр кода страницы» → найди
   `application/ld+json`: внутри три блока — Store, BreadcrumbList,
   ItemList с Product и offers (цены).
2. Скопируй содержимое блока в https://validator.schema.org
   («Фрагмент кода») — ошибок быть не должно.

После деплоя (на живом домене):
1. https://search.google.com/test/rich-results — URL страницы магазина.
   Должно найти: «Товары» (по числу товаров с ценами) и «Строки навигации».
2. https://pagespeed.web.dev — главная и магазин, смотреть Core Web Vitals.
3. Яндекс.Вебмастер → Инструменты → «Проверка микроразметки» — тот же URL.
4. Открыть `/sitemap.xml` и `/robots.txt` в браузере — оба отвечают.
5. Отправить ссылку на сайт себе в WhatsApp — развернётся карточка
   с картинкой и заголовком (это Open Graph в действии).

## Цели аналитики

Клики по «Позвонить», WhatsApp и «Перейти в магазин» (Kaspi) на страницах магазинов
отправляются событиями `call_click`, `whatsapp_click`, `kaspi_click` с параметром `shop`
(slug магазина) — это и есть «эффект» для решения по следующим этапам.

Отдельно трекается поиск: событие `search` (что ищут) и `search_empty`
(искали и не нашли — прямой сигнал, какого ассортимента на базаре не хватает
и каких арендаторов звать).

- Яндекс.Метрика: создать цели типа «JavaScript-событие» с этими идентификаторами
  (`call_click`, `whatsapp_click`, `kaspi_click`, при желании `search_empty`).
  Разрез по магазинам — через параметры визитов (`shop`).
- GA4: события приходят автоматически, при желании пометить их как ключевые (key events).
- Без заполненных `NEXT_PUBLIC_YM_ID` / `NEXT_PUBLIC_GA_ID` трекер молчит.
